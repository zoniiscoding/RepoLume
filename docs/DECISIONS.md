# RepoLume Architecture Decision Log

Decisions are append-only. If a decision changes, add a superseding entry instead of silently rewriting history.

## D-001 — Enforce milestone-by-milestone delivery

- **Date:** 2026-07-16
- **Status:** Accepted
- **Decision:** Implement only the milestone explicitly authorized by the user. Milestone 0 creates assessment and documentation only; Milestone 1 requires separate authorization.
- **Rationale:** The repository is greenfield and the specification explicitly forbids an uncontrolled whole-system pass.
- **Consequence:** Documents may describe later components, but they must label them planned rather than implemented.

## D-002 — Use the specified monorepo and deployment shape

- **Date:** 2026-07-16
- **Status:** Accepted
- **Decision:** Plan separate `backend`, `embedding_service`, and `frontend` packages, deployed to Vercel/Railway with Neon, private Redis, and Qdrant Cloud.
- **Rationale:** No existing repository alternative exists, and separate public/private services reinforce the trust boundary.
- **Consequence:** Milestone 1 will scaffold only the backend foundation needed by its acceptance criteria; later package functionality remains gated.

## D-003 — Use SQLAlchemy asynchronous sessions in API and worker

- **Date:** 2026-07-16
- **Status:** Accepted for Milestone 1 implementation
- **Decision:** Use SQLAlchemy 2.x `AsyncSession` with `asyncpg`. Create short-lived sessions per request or explicit worker transaction, never across slow network/clone/model operations.
- **Rationale:** One strategy across API and ARQ avoids sync/async duplication while keeping transactions bounded.
- **Consequence:** Queue payloads carry scalar IDs; workers reload durable state. Alembic remains synchronous if its migration runner is simpler, while application access stays async.

## D-004 — Use PostgreSQL as durable job truth and ARQ for delivery

- **Date:** 2026-07-16
- **Status:** Accepted for Milestone 3 implementation
- **Decision:** PostgreSQL owns job status, attempts, progress, heartbeats, and terminal errors. ARQ/Redis delivers job identifiers and supports retry scheduling.
- **Rationale:** Redis delivery alone does not satisfy restart recovery or auditability.
- **Consequence:** Enqueueing and recovery require idempotency and reconciliation; a Redis outage cannot erase durable job state.

## D-005 — Use UUIDv4 application IDs and unique external IDs

- **Date:** 2026-07-16
- **Status:** Accepted for Milestone 1 implementation
- **Decision:** Use UUIDv4 primary keys generated by the application, and store GitHub numeric identifiers in indexed, appropriately unique columns.
- **Rationale:** UUIDv4 is supported without extra runtime dependencies on Python 3.11 and does not expose record counts.
- **Consequence:** API identifiers do not prove authorization; every lookup remains tenant-aware.

## D-006 — Make authorization an application/persistence invariant

- **Date:** 2026-07-16
- **Status:** Accepted
- **Decision:** Protected services obtain repositories and sessions only through authorization-aware queries that include the authenticated user, active membership, installation state, selected repository, and parent relationships.
- **Rationale:** Scattered route-level ownership checks are easy to omit and a `repo_id` alone is explicitly insufficient.
- **Consequence:** Infrastructure and service APIs accept an authenticated actor context; model arguments never control repository scope.

## D-007 — Never execute connected repository code

- **Date:** 2026-07-16
- **Status:** Accepted, non-negotiable
- **Decision:** Treat repository bytes and all GitHub text as inert, untrusted data. Static parsing is allowed; execution, import, evaluation, installation, build, tests, hooks, plugins, and embedded commands are forbidden.
- **Rationale:** Repository access crosses the product's primary security boundary.
- **Consequence:** Clone tooling uses fixed allowlisted arguments and isolated temporary storage. RepoLume's own test suite is still executed normally.

## D-008 — Use application-owned provider abstractions without a heavy agent framework

- **Date:** 2026-07-16
- **Status:** Accepted for later milestones
- **Decision:** Define small embedding, vector-store, and LLM protocols and implement the tool loop directly.
- **Rationale:** This keeps providers replaceable, tenant enforcement visible, and the tool surface limited.
- **Consequence:** No LangChain, CrewAI, or AutoGen dependency. Provider behavior is validated at adapter boundaries.

## D-009 — Use inactive index versions and explicit activation

- **Date:** 2026-07-16
- **Status:** Accepted for Milestone 5 implementation
- **Decision:** Build vectors, symbols, and edges under a new inactive version and update the repository's active version only after all required stages succeed.
- **Rationale:** Progressive overwrites could expose partial or inconsistent evidence.
- **Consequence:** Cleanup and reconciliation are idempotent because PostgreSQL and Qdrant cannot share one atomic transaction.

## D-010 — Use structured content-minimizing observability

- **Date:** 2026-07-16
- **Status:** Accepted for Milestone 1 implementation
- **Decision:** Emit structured operational identifiers, timings, states, counts, and safe error codes. Exclude full prompts, messages, responses, chunks, repository contents, tokens, cookies, and secrets.
- **Rationale:** Operational visibility must not create a second repository-content store or a credential leak.
- **Consequence:** Debugging procedures rely on correlation IDs and safe error categories rather than raw private payloads.

## D-011 — Keep deletion asynchronous and verifiable

- **Date:** 2026-07-16
- **Status:** Accepted in principle; retention details deferred
- **Decision:** Block access immediately, then durably purge all repository-scoped vectors, graphs, chats, caches, jobs, and temporary artifacts before reporting completion.
- **Rationale:** A soft-delete flag does not meet the product's privacy requirement.
- **Consequence:** Exact operational-retention treatment for job and usage metadata must be finalized before Milestone 9 implementation.

## D-012 — Do not invent evaluation or production evidence

- **Date:** 2026-07-16
- **Status:** Accepted
- **Decision:** Record methodology before measurements and add results only from executed evaluation/deployment checks.
- **Rationale:** Trustworthiness is a product feature; fabricated benchmarks or readiness claims undermine it.
- **Consequence:** `docs/EVALUATION.md` currently has no scores, and Milestone 0 is explicitly not production-ready.

## D-013 — Standardize Milestone 1 on Python 3.13 and PostgreSQL 18

- **Date:** 2026-07-16
- **Status:** Accepted and implemented
- **Decision:** Use Python 3.13 as the production/CI baseline while retaining declared compatibility with Python 3.11–3.14. Use PostgreSQL 18 for local and CI foundation verification; never substitute SQLite.
- **Rationale:** One explicit production baseline makes locks, compiled drivers, CI, and container evidence reproducible while preserving the specification's supported Python range.
- **Consequence:** `.python-version`, GitHub Actions, and the container select Python 3.13. The initial migration and integration suite were verified on PostgreSQL 18.4.

## D-014 — Lock direct and transitive Python dependencies with hashes

- **Date:** 2026-07-16
- **Status:** Accepted and implemented
- **Decision:** Keep exact direct requirements in `backend/pyproject.toml` and commit separate production and development `pip-tools` lockfiles with hashes.
- **Rationale:** Exact, hashed resolution makes local, CI, and container installation converge and gives dependency audit a stable input.
- **Consequence:** Dependency updates require regenerating both locks under Python 3.13 and rerunning format, lint, type, migration, test, audit, and image checks.

## D-015 — Use constrained strings for persisted workflow states

- **Date:** 2026-07-16
- **Status:** Accepted and implemented
- **Decision:** Persist status/category values as strings with SQLAlchemy enums configured to create PostgreSQL check constraints instead of native PostgreSQL enum types.
- **Rationale:** Check-constrained strings preserve database validation while making later state evolution and migrations less operationally rigid than native enum alteration.
- **Consequence:** Application enums remain the typed source and Alembic owns their database constraints. Adding a state still requires an explicit migration.

## D-016 — Make safe failures and content-minimizing logs infrastructure concerns

- **Date:** 2026-07-16
- **Status:** Accepted and implemented
- **Decision:** Validate or generate request IDs at the ASGI boundary, return one stable error envelope, strip request values from validation errors, hide internal exception messages, and log only allowlisted operational fields.
- **Rationale:** Routes should not have to remember redaction rules, and unsafe framework defaults can disclose untrusted values or internal exception text.
- **Consequence:** Uvicorn access logs are disabled, database URLs remain secret-valued, startup emits only a safe settings summary, and tests cover request IDs, validation sanitization, internal errors, and log exclusions.

## D-017 — Keep migrations deliberate and health dependency-specific

- **Date:** 2026-07-16
- **Status:** Accepted and implemented
- **Decision:** Do not run Alembic automatically during API startup. Liveness never probes dependencies; readiness performs only a bounded PostgreSQL check and fails with a safe 503 response.
- **Rationale:** Automatic migrations create unsafe multi-instance rollout races, while dependency-coupled liveness can cause restart loops during a database outage.
- **Consequence:** Operators and CI run `alembic upgrade head` explicitly before compatible application rollout. Readiness can remove an instance from service without declaring its process dead.

## D-018 — Establish least-privilege CI and a non-root container baseline

- **Date:** 2026-07-16
- **Status:** Accepted and implemented
- **Decision:** Give GitHub Actions read-only repository contents permission, disable persisted checkout credentials, install only hashed locks, verify migrations against PostgreSQL 18, audit production dependencies, and separately build/inspect a container configured as UID/GID `10001:10001`.
- **Rationale:** Foundation CI should prove reproducibility and basic supply-chain/container posture before deployment automation exists.
- **Consequence:** The workflow has no deployment permission or secrets. Local Podman verification reproduced the image build, non-root configuration, startup, and health checks; GitHub-hosted execution remains pending the first remote run.
